Architecture Approaches

Three ways to structure the HanahOne Group ERP

A

Single Multi-Tenant Database

Recommended

One database with a company_id column on every table. All three companies share the same schema. Consolidated group reports are simple JOINs. Inter-company transfers are rows in the same DB.

Pros

  • Simplest to build and maintain
  • Consolidated reporting is trivial
  • Inter-company transfers are easy
  • Single deployment, single codebase
  • Perfect for ~1000 SKUs / 5+ users

Cons

  • Must enforce data isolation via app logic
  • Schema changes affect all companies
  • Company-specific customization is harder
B

Separate Databases per Company

Each company gets its own database. A consolidation layer aggregates data for group-level views.

Pros

  • Strong data isolation by default
  • Each company can evolve independently
  • Easier to split off a company later

Cons

  • Consolidated reporting requires cross-DB queries or ETL
  • Inter-company transfers are complex (distributed transactions)
  • 3x the operational overhead
  • Overkill for this scale
C

Microservices per Domain

Separate services for Sales, Inventory, Manufacturing, CRM, etc. Each service owns its data. Companies are a dimension within each service.

Pros

  • Clean domain boundaries
  • Independent scaling per domain
  • Team independence

Cons

  • Massive overengineering for 5 users / 1000 SKUs
  • Distributed system complexity
  • Much slower to build
  • Harder to maintain with a small team
